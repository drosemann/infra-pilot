# feature 42: internationalization (i18n) & localization (l10n)

- feature id: 42
- primary service: management panel
- effort: medium (4-6 pt)
- phase: phase 5 (ux, mobile & compliance)
- dependencies: management panel v1 (react/vite)

## overview

full internationalization (i18n) and localization (l10n) support for the management panel web application. enables the platform to serve users in their preferred language with culturally appropriate formatting (dates, numbers, currencies) and layout direction (ltr/rtl).

a crowdin-based translation workflow allows both professional translators and community contributors to submit and review translations. the i18n infrastructure is designed to be shared across the entire frontend ecosystem (panel, desktop app, and future mobile app).

### key capabilities

- locale detection — automatic detection from browser, saved preference, url override
- translation files — json-based message catalogs with namespaced organization
- crowdin integration — ota translation delivery, community contribution workflow
- rtl support — full bidirectional layout with css logical properties
- pluralization — icu messageformat plurals, ordinals, and select rules
- interpolation — variable substitution, rich text formatting, components in translations
- date/time/number — locale-aware formatting via intl api
- fallback chains — `de-AT` → `de` → `en` locale fallback with missing-key warnings
- lazy loading — code-split translation chunks, load on demand per locale
- accessibility — `lang` attribute, `dir` attribute, translated aria labels

### technology stack

| layer | technology | rationale |
|---|---|---|
| i18n framework | react-intl (formatjs) | industry standard, icu messageformat, rich text components |
| build integration | @formatjs/cli + vite-plugin-react-intl | extract messages at build time, type-safe message ids |
| translation platform | crowdin enterprise | ota delivery, community contributions, glossaries, tm |
| rtl | css logical properties + postcss-rtlcss | automated rtl css transforms, no manual styles |
| locale detection | @formatjs/intl-localematcher + negotiator | rfc 4647 language matching, quality values |
| date/time/number | @formatjs/intl (polyfill) | consistent formatting across browsers, locale-aware |
| pluralization | icu selectformat | built into react-intl, handles cldr plural rules |

**provider setup:**

```tsx
// i18n/provider.tsx
import { IntlProvider } from 'react-intl';
import { useLocale } from './useLocale';
import { loadMessages } from './loader';

export function I18nProvider({ children }: { children: React.ReactNode }) {
  const { locale, messages, direction } = useLocale();

  if (!messages) {
    return <LoadingFallback />;  // Skeleton while translations load
  }

  return (
    <IntlProvider
      locale={locale}
      messages={messages}
      defaultLocale="en"
      onError={(err) => {
        if (process.env.NODE_ENV === 'development') {
          console.warn('[i18n]', err.message);
        }
      }}
    >
      <div dir={direction} lang={locale}>
        {children}
      </div>
    </IntlProvider>
  );
}
```

**translation loader:**

```ts
// i18n/loader.ts
const messageLoaders: Record<string, () => Promise<Record<string, any>>> = {
  en: () => import('../locales/en/common.json').then(m => m.default),
  de: () => import('../locales/de/common.json').then(m => m.default),
  fr: () => import('../locales/fr/common.json').then(m => m.default),
  // ...
};

export async function loadMessages(locale: string) {
  const loader = messageLoaders[locale];
  if (!loader) {
    // Fall back to parent locale (de-AT → de → en)
    const base = locale.split('-')[0];
    const fallback = messageLoaders[base] || messageLoaders.en;
    return fallback();
  }
  return loader();
}
```

**custom hook for convenience:**

```tsx
// i18n/useT.ts
import { useIntl } from 'react-intl';

export function useT() {
  const intl = useIntl();

  return {
    t: (id: string, values?: Record<string, any>) => intl.formatMessage({ id }, values),
    locale: intl.locale,
    direction: intl.locale === 'ar' || intl.locale === 'he' ? 'rtl' as const : 'ltr' as const,
    formatDate: intl.formatDate,
    formatNumber: intl.formatNumber,
    formatRelativeTime: intl.formatRelativeTime,
  };
}
```

## api design

no new backend endpoints are required. the i18n system is entirely frontend-local.

however, the following may be added for backend-driven locale preference storage:

```yaml
# User locale preference (optional — can be purely frontend/localStorage)
GET /api/v2/user/preferences
  Response:
    locale: "de"
    timezone: "Europe/Berlin"

PUT /api/v2/user/preferences
  Body:
    locale: "fr"
    timezone: "Europe/Paris"
  Response:
    success: true

# Server-side locale detection (for emails, notifications)
GET /api/v2/user
  Response:
    locale: "de"   # Used for transactional emails
    ...
```

## data model

no new database tables. locale preference is stored:

- primary: `localStorage` key `infrapilot_locale`
- secondary (if api used): `user_preferences.locale` column
- url override: `?lang=de` query parameter (highest priority)

```sql
-- Optional: if backend stores locale preference
ALTER TABLE user_preferences ADD COLUMN locale TEXT DEFAULT 'en';
```

## rtl implementation checklist

| component area | action |
|---|---|
| navigation sidebar | use `margin-inline-end` instead of `margin-left` |

## shared i18n library (ecosystem strategy)

the translation files and i18n infrastructure should be extracted into a shared package to avoid duplication across panel, desktop, and mobile:

```
packages/
└── i18n/
    ├── src/
    │   ├── locales/         # JSON translation files
    │   ├── index.ts         # Public API
    │   ├── useT.ts          # React hook
    │   ├── loader.ts        # Dynamic import loader
    │   └── locales.ts       # Locale config
    ├── crowdin.yml
    └── package.json
```

consumed by:
- `management-panel` (react) — direct npm dependency
- `desktop-app` (tauri + react) — same npm dependency
- `mobile-app` (flutter) — needs separate dart/arb files, but shares crowdin project
